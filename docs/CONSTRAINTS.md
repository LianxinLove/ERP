# ERP系统开发约束文档

本文档记录ERP系统开发过程中的所有约束和规范，必须在每次开发前阅读确认。

---

## 一、项目概述约束

### 1.1 项目定位
- 这是一个**展示AI工具开发能力**的项目
- 核心重点是**技术栈的深度使用**
- 必须体现**高代码质量和工程化能力**

### 1.2 技术栈（不可更改）
- **前端框架**：React、Next.js 14 (App Router)、TypeScript
- **UI组件库**：shadcn/ui（基于 Radix UI）
- **样式方案**：Tailwind CSS
- **状态管理**：Zustand + React Context
- **数据获取**：React Query (TanStack Query)
- **表单处理**：React Hook Form + Zod
- **后端框架**：Python、FastAPI
- **数据库**：MySQL 8.0+
- **认证方式**：JWT (Access + Refresh Token)

---

## 二、代码质量约束

### 2.1 代码风格
- **清晰简洁**：避免过度设计，保持代码可读性
- **高可维护性**：模块化设计，低耦合高内聚
- **高扩展性**：预留扩展点，使用工厂模式/策略模式

### 2.2 注释规范
- **中文注释**：所有注释使用简体中文
- **公共组件**：必须包含完整的JSDoc注释
- **复杂逻辑**：必须包含详细说明

**注释模板**：
```typescript
/**
 * [组件/函数简要描述]
 * 
 * @description [详细描述功能]
 * 
 * @features [列出主要特性]
 * - 特性1
 * - 特性2
 * 
 * @param xxx - [参数说明]
 * @returns [返回值说明]
 * 
 * @example
 * ```tsx
 * <Component prop="value" />
 * ```
 */
```

### 2.3 命名规范
- **前端**：
  - 组件：PascalCase (UserProfile.tsx)
  - Hook：camelCase with 'use' prefix (useUser.ts)
  - 工具函数：camelCase (formatDate.ts)
  - 常量：UPPER_SNAKE_CASE (API_BASE_URL)
  - 接口/类型：PascalCase with 'I' prefix (IUser)

- **后端**：
  - 类：PascalCase (UserService)
  - 函数/变量：snake_case (get_user)
  - 常量：UPPER_SNAKE_CASE (MAX_RETRY)
  - 私有成员：单下划线前缀 (_private_method)

### 2.4 错误边界约束
- **每个页面级组件**必须包裹 ErrorBoundary
- **每个异步操作**必须包含错误处理
- **API调用**必须有统一的错误拦截
- **用户可见的错误**必须有友好的中文提示

**错误处理模板**：
```typescript
try {
  // 业务逻辑
} catch (error) {
  // 1. 记录错误日志
  logger.error('操作失败', error);
  
  // 2. 用户提示
  toast.error('操作失败，请稍后重试');
  
  // 3. 错误上报（可选）
  // reportError(error);
}
```

## 三、架构约束

### 3.1 分层架构
```
┌─────────────────────────────────────┐
│           前端 (Next.js)            │
├─────────────────────────────────────┤
│  UI层 │ 状态层 │ 业务逻辑层 │ API层 │
└─────────────────────────────────────┘
                 ↕ HTTP
┌─────────────────────────────────────┐
│           后端 (FastAPI)            │
├─────────────────────────────────────┤
│ API层 │ 服务层 │ 数据访问层 │ 数据库 │
└─────────────────────────────────────┘
```

### 3.2 模块边界
- **前端模块间**：通过共享类型和API通信
- **后端模块间**：通过服务层接口调用
- **禁止**：跨模块直接访问内部实现

### 3.3 共享代码约束
- 类型定义必须放在 `packages/types`
- UI组件必须放在 `packages/ui` 或各应用的 `components/ui`
- 工具函数优先放在共享包

## 四、开发流程约束

### 4.1 开发前检查
每次开始开发前必须：
1. **阅读记忆文档**：确认当前进度和需求
2. **查看 task_plan.md**：确认当前阶段和任务
3. **查看 findings.md**：确认技术决策
4. **查看 progress.md**：确认之前的工作

### 4.2 开发规范
- **先设计后编码**：复杂功能先写设计文档
- **小步提交**：每个功能点完成后立即提交
- **测试驱动**：关键功能必须写测试

### 4.3 上下文管理
- **80%规则**：当上下文接近80%容量时
  1. 保存当前进度到 progress.md
  2. 完成当前任务
  3. 压缩上下文（清除已完成的非关键信息）
  4. 开始新会话继续下一阶段

### 4.4 模块化执行
- **独立子代理**：每个模块/阶段由独立子代理执行
- **任务分解**：大任务必须拆解为小任务
- **进度同步**：每个子任务完成后更新文档

## 五、功能模块约束

### 5.1 账号系统
- 必须支持JWT认证
- 必须支持Token自动刷新
- 必须支持多设备登录控制
- 密码必须使用bcrypt加密

### 5.2 权限系统
- 采用RBAC模型
- 前后端双重权限检查
- 权限必须可配置
- 权限检查必须有缓存

### 5.3 审批流程
- 必须支持可视化流程设计
- 必须支持串行/并行/条件分支
- 必须支持退回/转办/加签
- 流程实例状态必须可追溯

### 5.4 ERP模块
- 所有列表必须支持分页、排序、筛选
- 所有操作必须记录操作日志
- 关键操作必须有审批流程
- 数据删除必须是软删除

---

## 六、UI/UX约束

### 6.1 设计原则
- 使用 **UI-UX-Pro-Max skill** 进行设计评审
- 统一的设计语言
- 一致的交互模式

### 6.2 响应式设计
- 支持 1920px、1440px、1024px、768px 断点
- 关键操作在移动端可用

### 6.3 可访问性
- 语义化HTML
- 键盘导航支持
- 屏幕阅读器友好

---

## 七、性能约束

### 7.1 前端性能
- 首屏加载 < 2秒
- 路由切换 < 500ms
- 使用React Query缓存
- 图片懒加载

### 7.2 后端性能
- API响应 < 500ms (P95)
- 数据库查询优化
- 使用连接池
- 关键接口使用缓存

---

## 八、安全约束

### 8.1 认证安全
- Token存储在HttpOnly Cookie或LocalStorage
- 敏感操作需要二次确认
- 登录失败次数限制

### 8.2 数据安全
- 敏感数据加密存储
- API输入验证
- SQL注入防护（使用ORM）
- XSS防护

### 8.3 权限安全
- 前端权限只是UI控制
- 后端必须进行权限验证
- 默认拒绝原则

---

## 九、测试约束

### 9.1 测试覆盖
- 核心业务逻辑：单元测试覆盖率 > 80%
- 关键API：集成测试必须
- 主要用户流程：E2E测试必须

### 9.2 测试类型
- 单元测试：pytest (Python) + Jest (TS)
- 集成测试：pytest + FastAPI TestClient
- E2E测试：Playwright

## 十一、Git规范

### 11.1 分支策略
- main：生产环境
- develop：开发环境
- feature/*：功能分支
- bugfix/*：修复分支

### 11.2 提交规范
```
<type>(<scope>): <subject>

type: feat | fix | docs | style | refactor | test | chore
scope: module-name
subject: 简短描述（中文）

示例：
feat(auth): 添加JWT认证功能
fix(user): 修复用户编辑页面bug
docs(api): 更新API文档
```

## 十二、禁止事项

❌ **禁止硬编码**：配置必须从环境变量读取
❌ **禁止魔法数字**：使用常量定义
❌ **禁止console.log**：使用日志库
❌ **禁止any类型**：TypeScript项目必须明确定义类型
❌ **禁止直接操作DOM**：使用React声明式API
❌ **禁止提交敏感信息**：密码、Token、密钥
❌ **禁止复制粘贴代码**：抽取公共函数/组件
❌ **禁止绕过权限检查**：前后端都必须验证
